# Internationalization (i18n)

civic-pulse ships with a tiny, dependency-free i18n layer so the same React app can run in any language. The framework landed in v0.1; ongoing string extraction is a gradual contribution effort.

## How it works

```
locales/<code>.json                 # the dictionaries (en, sq, de, …)
        ↓
src/i18n.ts                         # selects the dictionary based on country config
        ↓
@core/i18n   I18nProvider           # wraps the React tree in src/index.tsx
        ↓
useT() / useMonthNames()            # hooks consumed by every component
```

The country chooses its locale in `countries/<code>/config.json`:

```json
{ "locale": "sq" }
```

At build time, `src/i18n.ts` loads `locales/<requested-locale>.json` as the active dictionary and `locales/en.json` as the fallback. Missing keys fall back to English; if a key is missing from both, the key itself is rendered (so a regression is visible, not silent).

## Adding a new locale

1. Copy `locales/en.json` to `locales/<code>.json` (e.g. `locales/fr.json`).
2. Translate every value. Keep keys identical.
3. Reference the new locale from your country config: `"locale": "fr"`.
4. Run `COUNTRY=<your-code> npm run build` — the glob in `i18n.ts` picks it up automatically.

You don't have to translate every key. Anything missing from your locale falls back to `locales/en.json`. This makes incremental localization safe.

## Adding a new string

Until every string is extracted, you'll often add a new UI string before a key for it exists. The pattern:

1. **Add the key + English text to `locales/en.json`.** Keys are dot-namespaced by area: `nav.*`, `status.*`, `filter.*`, `stats.*`, `methodology.*`, etc. Add new namespaces freely.
2. **Translate it in every other locale you can.** At minimum add it to whatever locale your active country uses (set by `locale` in `countries/<code>/config.json`) so that production doesn't fall back.
3. **Use it in the component.**

```tsx
import { useT } from '@core/i18n';

const MyComponent = () => {
  const t = useT();
  return <button>{t('filter.search')}</button>;
};
```

For interpolation:

```jsonc
// locales/en.json
{ "deputy.activity": "{name} took the floor {count} times" }
```

```tsx
t('deputy.activity', { name: deputy.name, count: deputy.activity.speechCount })
```

## Migrating an existing hardcoded string

The current state: many strings inside `components/*.tsx` and parts of `App.tsx` are still hardcoded Albanian. To migrate one:

1. Find the literal: `"Të Gjitha"`, `"Si funksionon"`, etc.
2. Pick a key name in `locales/en.json`. Reuse existing namespaces where it fits.
3. Add the same key to `locales/sq.json` with the original Albanian string.
4. Replace the literal in the component with `t('...')`.
5. Run `COUNTRY=example npm run build` — output should be byte-for-byte identical aside from the new key reference.

## Reference implementation in this repo

See `App.tsx` for examples of:

- `useT()` for a one-off label.
- `useMonthNames()` for the locale's month names (used by `formatShortDate` / `formatMonthYear`).
- `buildStatusLabels(t)` for building a status → label map driven by the translator.

## Why not i18next / react-intl?

Both are excellent, but they ship with format machinery (ICU, plurals, gender, fallback chains) we don't need yet. Our needs are: pick a JSON file, look up a key, do `{var}` substitution. That's ~20 lines of code, so we keep it inline.

If we later need ICU pluralization or other format features, swap `createTranslator` in `packages/core/src/i18n.ts` for an i18next adapter — no component code changes needed.

## Coverage

As of v0.2, **every UI surface in the React app routes its strings through the i18n dictionary**:

- `App.tsx` (hero, filters, nav, header, footer, 404)
- `components/DashboardStats.tsx`, `PromiseCard.tsx`, `PromiseDetail.tsx`
- `components/DeputiesDirectory.tsx`, `DeputyProfile.tsx`
- `components/QeveriaOrgChart.tsx`, `QeveriaMinisterDetail.tsx`
- `components/RadarSpiderChart.tsx`
- `components/StaticPage.tsx` (Methodology / About / Contact / Privacy bodies live per-country in `countries/<code>/pages/*.json`)

A new country can change `config.json.locale` and add `locales/<code>.json` — no React or component edits needed.

Number and date formatting uses `Intl` with the country's `dateLocale` (from `config.json`) in every migrated component.
